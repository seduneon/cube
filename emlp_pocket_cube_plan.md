# EMLP on the 2×2×2 Rubik's Cube — Experiment Plan

## Goal

Train an equivariant MLP (EMLP) to solve the 2×2×2 Rubik's Cube (Pocket Cube) and compare it against a standard MLP baseline. The experiment demonstrates how baking the cube's spatial rotation symmetry into the network architecture improves data efficiency, parameter efficiency, and generalization.

## Background

The 2×2×2 has 3,674,160 reachable states and God's number of 11 (HTM) / 14 (QTM). The spatial rotation group of a cube has 24 elements. Two cube states related by a whole-cube rotation are equally hard to solve — this is the symmetry we exploit.

The `emlp` package (Finzi et al., ICML 2021) provides automated construction of equivariant layers for arbitrary matrix groups. It is JAX-based (with objax). We define the 24-element rotation group acting on the 24 sticker positions and let EMLP compute the equivariant weight bases.

## Phase 1 — Group and Representation Setup

### File: `cube_group.py`

**Objective:** Define the 24-element rotation group of the cube as an EMLP `Group`, acting on the 24 sticker positions of the 2×2×2.

**Details:**

1. **Sticker labeling convention.** The 2×2×2 has 6 faces × 4 stickers = 24 stickers. Number them 0–23:
   - Face U (up): stickers 0, 1, 2, 3
   - Face D (down): stickers 4, 5, 6, 7
   - Face F (front): stickers 8, 9, 10, 11
   - Face B (back): stickers 12, 13, 14, 15
   - Face L (left): stickers 16, 17, 18, 19
   - Face R (right): stickers 20, 21, 22, 23
   
   Within each face, index the 4 stickers in reading order (top-left, top-right, bottom-left, bottom-right when looking at that face).

2. **Rotation group generators.** The 24-element rotation group O of the cube is generated by two rotations:
   - `x_rot`: 90° rotation about the axis through the R and L face centers (R face clockwise when viewed from R side). This cycles U→F→D→B and permutes stickers within those faces accordingly.
   - `y_rot`: 90° rotation about the axis through the U and D face centers (U face clockwise when viewed from top). This cycles F→L→B→R.
   
   Express each as a permutation of the 24 sticker indices and convert to a 24×24 permutation matrix.

3. **EMLP Group subclass:**
   ```python
   from emlp.groups import Group
   import numpy as np

   class CubeRotationGroup(Group):
       """The 24-element rotation group of the cube acting on 24 sticker positions."""
       def __init__(self):
           # x_rot_perm and y_rot_perm are length-24 arrays mapping old index -> new index
           self.discrete_generators = np.stack([
               perm_to_matrix(x_rot_perm),
               perm_to_matrix(y_rot_perm),
           ])  # shape (2, 24, 24)
           super().__init__(n=24)
   ```
   where `perm_to_matrix(perm)` converts a permutation array to a permutation matrix.

4. **Verification steps:**
   - Enumerate all 24 group elements by multiplying generators and confirm exactly 24 distinct matrices.
   - Check that the group action on sticker indices is consistent: applying `y_rot` four times returns to identity, `x_rot` four times returns to identity, and the group has the correct structure.
   - Compute `V(CubeRotationGroup()).equivariant_basis()` and print its shape — this tells us the rank of the equivariant subspace for linear maps on the 24-dim representation.

## Phase 2 — Cube Environment and Dataset Generation

### File: `cube_env.py`

**Objective:** Implement a minimal 2×2×2 cube simulator.

**Details:**

1. **State representation.** A cube state is a length-24 array where `state[i]` is the color (0–5) of sticker position `i`. The solved state has `state[i] = i // 4` (stickers 0–3 are color 0, stickers 4–7 are color 1, etc.).

2. **Moves.** Implement the 3 generators (fixing the BLD corner) and their inverses, for 6 moves total:
   - `R` (right face clockwise), `R'` (right face counter-clockwise)
   - `U` (up face clockwise), `U'`
   - `F` (front face clockwise), `F'`
   
   Each move is a permutation of the 24 sticker indices. Define these using the same sticker labeling as Phase 1. The SymPy reference for 2×2×2 generators can be used as a cross-check:
   ```
   F = (2,19,21,8)(3,17,20,10)(4,6,7,5)  # SymPy 0-indexed convention
   R = (1,5,21,14)(3,7,23,12)(8,10,11,9)
   D = (6,18,14,10)(7,19,15,11)(20,22,23,21)
   ```
   **Important:** Verify our sticker labeling matches or adapt these permutations to our convention. The exact permutation cycles depend on the labeling. Test that applying R four times returns to the solved state.

3. **State encoding for neural network.** One-hot encode each sticker's color:
   ```python
   def encode_state(state):
       """state: (24,) int array with values 0-5 -> (144,) float one-hot vector"""
       return np.eye(6)[state].flatten()  # shape (144,)
   ```

4. **Random scramble generation:**
   ```python
   def scramble(state, n_moves):
       """Apply n_moves random moves, return (new_state, list_of_moves)."""
   ```

### File: `dataset.py`

**Objective:** Generate training and test data with exact optimal distances.

**Details:**

1. **BFS for exact distances.** Run BFS from the solved state over the full state space. This requires a canonical state representation for hashing. Use the state array as a tuple for dict keys.
   - Expected: visits all 3,674,160 states
   - Stores: `{state_tuple: optimal_distance}` dictionary
   - Memory: ~300 MB (can be reduced with more compact encoding if needed)
   - Runtime: ~5–10 minutes on CPU
   - Also store the optimal first move for each state (for policy ground truth): the move that leads to a neighbor with distance `d-1`.

2. **Training set generation (backward scrambles):**
   ```python
   def generate_training_data(n_samples, bfs_table):
       """
       For each sample:
       1. Start from solved state
       2. Apply k random moves (k ~ Uniform(1, 14))
       3. Record: (encoded_state, true_bfs_distance, optimal_move_label)
       """
   ```
   Use the BFS table to look up the true distance (not the scramble depth, which is an upper bound).

3. **Dataset sizes:**
   - Training: 50,000 / 200,000 / 1,000,000 samples
   - Validation: 20,000 samples (separate scrambles)
   - Test: 10,000 samples, stratified by depth (roughly uniform across depths 1–11)

4. **Output format:** NumPy arrays saved to disk:
   - `X_train.npy`: shape `(N, 144)` float32
   - `y_distance_train.npy`: shape `(N,)` int — optimal distance (0–11)
   - `y_policy_train.npy`: shape `(N,)` int — optimal move index (0–5)
   - Same for val and test splits

## Phase 3 — Model Construction

### File: `models.py`

**Objective:** Define the EMLP and MLP models with matched architecture.

**Details:**

1. **Input/output representations for EMLP:**
   - Input: The 144-dim one-hot vector. This is 24 copies of a 6-dim vector (one per sticker). We represent this as `6*V` where `V` is the base 24-dim permutation representation of `CubeRotationGroup`. When the group permutes sticker positions, the 6-dim one-hot blocks follow accordingly. Alternatively, this can be expressed as `V ⊗ ℝ⁶` — the tensor product of the permutation rep with a trivial 6-dim space. In EMLP syntax: `repin = 6*V` where `V = emlp.reps.V(G)`.
   - Value output: `Scalar` (1-dim, invariant under rotations). The distance to solved doesn't change under whole-cube rotation.
   - Policy output: This is trickier. The 6 moves {R, U, F, R', U', F'} don't form a standard representation of the rotation group. For simplicity, **train two separate models**: one EMLP for the value head (invariant output) and one standard MLP for the policy head. The value head is where equivariance gives the clearest benefit.

   **Revised plan:** Focus the EMLP vs MLP comparison on the **value network** (distance prediction). This is the cleaner experiment because:
   - The output is a scalar (trivially invariant)
   - The input representation is well-defined
   - The metric (distance MAE) is unambiguous
   - We can still use the value network for solving via greedy search (pick the move whose resulting state has the lowest predicted distance)

2. **EMLP model:**
   ```python
   import emlp.nn as nn
   from emlp.reps import V, Scalar

   G = CubeRotationGroup()
   repin = 6 * V(G)    # 144-dim input
   repout = Scalar(G)   # 1-dim invariant output

   emlp_model = nn.EMLP(repin, repout, group=G, ch=384, num_layers=3)
   ```

3. **MLP baseline:**
   ```python
   mlp_model = nn.MLP(repin, repout, group=G, ch=384, num_layers=3)
   ```
   This is EMLP's built-in unconstrained MLP. Same hidden dims, same depth, same nonlinearities — but no equivariance constraints. The `group` argument is only used for input/output shapes, not for constraining weights.

4. **Report parameter counts:**
   ```python
   n_params_emlp = sum(p.value.size for p in emlp_model.vars())
   n_params_mlp = sum(p.value.size for p in mlp_model.vars())
   print(f"EMLP: {n_params_emlp:,} params, MLP: {n_params_mlp:,} params")
   ```

## Phase 4 — Training

### File: `train.py`

**Objective:** Train both models with identical hyperparameters.

**Details:**

1. **Training setup:**
   ```
   Optimizer:     Adam
   Learning rate: 3e-4 with cosine decay to 1e-5
   Batch size:    256
   Max epochs:    100
   Early stop:    patience=10 on validation MSE
   Loss:          MSE on predicted distance vs true BFS distance
   Seeds:         3 independent runs per configuration (seeds 0, 1, 2)
   ```

2. **Training loop** (standard EMLP/objax pattern):
   ```python
   import objax
   import jax.numpy as jnp

   @objax.Function.with_vars(model.vars())
   def loss_fn(x, y):
       pred = model(x)
       return jnp.mean((pred.squeeze() - y) ** 2)

   grad_fn = objax.GradValues(loss_fn, model.vars())
   opt = objax.optimizer.Adam(model.vars())

   @objax.Function.with_vars(model.vars() + opt.vars())
   def train_step(x, y, lr):
       g, v = grad_fn(x, y)
       opt(lr=lr, grads=g)
       return v
   ```

3. **Logging per epoch:**
   - Train MSE, Val MSE
   - Learning rate
   - Wall-clock time

4. **Configurations to train (6 total per seed = 18 runs):**

   | Model | Train size | Label |
   |-------|-----------|-------|
   | EMLP  | 50K       | emlp_50k |
   | EMLP  | 200K      | emlp_200k |
   | EMLP  | 1M        | emlp_1m |
   | MLP   | 50K       | mlp_50k |
   | MLP   | 200K      | mlp_200k |
   | MLP   | 1M        | mlp_1m |

5. **Checkpointing:** Save best model (by val MSE) for each configuration. Save as numpy `.npz` of all model variables.

## Phase 5 — Validation

### File: `evaluate.py`

**Objective:** Comprehensive evaluation of both models.

**Details:**

### Metric 1: Equivariance Error

For a random batch of 500 test states:
```python
def equivariance_error(model, states, group):
    """
    For each state x and random rotation g:
      err = |f(g·x) - f(x)|   (for invariant value head, these should be equal)
    Return mean absolute error.
    """
```
- Sample 10 random rotations from the 24-element group
- Compute mean |v(g·x) - v(x)| over states and rotations
- EMLP should give ~1e-7 (float32 precision), MLP will be larger

### Metric 2: Value Prediction Accuracy

On the full 10K test set:
- **MAE** (mean absolute error) of predicted distance vs true BFS distance
- **Per-depth MAE**: break down by true distance d = 0, 1, 2, ..., 11
- **Correlation**: Pearson r between predicted and true distance
- **Rounded accuracy**: fraction of predictions where `round(pred) == true_distance`

### Metric 3: Greedy Solve Rate

```python
def greedy_solve(model, state, max_steps=50):
    """
    At each step:
    1. For each of the 6 possible moves, compute the resulting state
    2. Predict value (distance) for each resulting state
    3. Pick the move with the lowest predicted distance
    4. If predicted distance ≈ 0, check if actually solved
    Return: (solved: bool, n_moves: int, move_sequence: list)
    """
```
- Test on 1000 random scrambles at depths 4, 7, 10, 14 (QTM)
- Report: solve rate (%), mean solution length, mean excess moves vs optimal

### Metric 4: Data Efficiency Curve

- Plot val MSE vs training set size (50K, 200K, 1M) for both models
- Key question: at what dataset size does MLP match EMLP's performance with 50K samples?

## Expected Results

Based on the theory and prior work:

- **Parameter count**: EMLP should have roughly 10–24× fewer effective parameters than MLP (depending on the equivariant basis rank for the rotation group on this representation).
- **Data efficiency**: EMLP trained on 50K samples should match or beat MLP trained on 200K–1M samples in value prediction MAE.
- **Equivariance error**: EMLP ≈ 1e-7, MLP ≈ 0.1–1.0 (depending on training data size).
- **Solve rate**: Both models should achieve >90% solve rate with greedy search when trained on sufficient data, but EMLP should reach this threshold with less data.
- **Per-depth accuracy**: The largest EMLP advantage should appear at depths 8–11 (the rarest states in the scramble distribution), where the MLP struggles from insufficient coverage of rotated variants.

## Important Notes for Implementation

1. **The rotation group, not the cube group.** We equivarify against the 24-element spatial rotation group O, NOT the 3.67M-element cube move group. The rotation group is the symmetry that relates equivalent puzzle states. The cube move group IS the state space — equivarifying against it would trivialize the problem.

2. **One-hot encoding alignment.** The one-hot encoding must be consistent with the group action. When rotation `g` permutes sticker positions, the 144-dim vector must permute the same way: the 6-dim block at position `i` moves to position `g(i)`. This is automatic if we use `6*V` as the input representation.

3. **BFS memory.** The BFS dictionary for 3.67M states is the memory bottleneck. Use tuple(state) as dict keys and int distances as values. If memory is tight, consider a more compact state hash (e.g., encoding the 24 sticker colors as a single integer in base 6).

4. **JAX/objax quirks.** EMLP uses objax, which has a different variable management pattern than PyTorch or pure JAX. Model parameters are in `model.vars()`. JIT compilation happens automatically. First epoch will be slow due to JAX tracing.

5. **Random seed discipline.** Set `np.random.seed()`, `jax.random.PRNGKey()`, and Python `random.seed()` identically for reproducibility. Use separate seeds for dataset generation vs training.
